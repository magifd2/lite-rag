# Request for Proposal (RFP): CLI-Based RAG Document Search System

## 1. Introduction

This document outlines the requirements for the development of a CLI-based document retrieval and question-answering system. The core of this system will be a Retrieval-Augmented Generation (RAG) architecture, leveraging a local Large Language Model (LLM) accessed via an OpenAI-compatible API endpoint.

The primary goal is to create a portable, efficient, and powerful tool for searching and synthesizing information from a set of Markdown documents.

## 2. Project Overview

The proposed system is a command-line application that allows users to:
1.  **Index** a collection of Markdown documents, creating a searchable vector database.
2.  **Query** the indexed documents with natural language questions and receive concise, context-aware answers generated by an LLM.

While the primary interface is a CLI, the system should be designed with the potential to serve as a backend for other frontends in the future.

## 3. Requirements

### 3.1. Functional Requirements

-   **Indexing (`index` command):**
    -   Accept a path to a directory of Markdown files.
    -   Parse the documents and divide them into meaningful semantic chunks.
    -   Generate vector embeddings for each chunk using the specified embedding model.
    -   Store the chunks and their corresponding vectors in a DuckDB database.
    -   The indexing process should be idempotent and can be re-run to update the database with new or modified documents.

-   **Question Answering (`ask` command):**
    -   Accept a natural language question from the user.
    -   Generate an embedding for the question.
    -   Perform a vector similarity search in DuckDB to retrieve the most relevant document chunks.
    -   Construct a prompt for the chat LLM, combining the user's question with the retrieved context.
    -   Send the prompt to the LLM and stream the generated answer back to the user's terminal.

### 3.2. Technical Stack

-   **Programming Language:** Go
-   **Database:** DuckDB (for both relational data and vector storage/search)
-   **LLM Backend:** LM Studio (Local Server, OpenAI API compatible)
-   **Embedding Model:** `nomic-ai/nomic-embed-text-v1.5-GGUF`
-   **Chat Model:** `openai/gpt-oss-20b` (pre-configured and available in the user's environment)
-   **CLI Framework:** A standard Go library like `cobra` is recommended.

### 3.3. Non-Functional Requirements

-   **Portability:** The final executable should be cross-compiled for `linux/amd64`, `linux/arm64`, `darwin/amd64`, and `darwin/arm64`.
-   **Performance:** Indexing and query response times should be reasonably fast for typical documentation sets.
-   **Configuration:** The application must support external configuration (e.g., via a file like `config.toml` or environment variables) for settings like API endpoints, model names, and database paths.
-   **Maintainability:** The codebase must adhere to the rules defined in `RULES.md`, promoting clarity, testability, and separation of concerns.

## 4. Known Issues & Challenges

During the initial setup phase, a significant challenge was encountered related to Go module dependencies.

-   **Problem:** The `go mod tidy` and `go get` commands failed to download modules (`github.com/duckdb/go-duckdb`), reporting a "Repository not found" error.
-   **Root Cause Analysis:** This issue is attributed to the development environment's security sandbox (macOS Seatbelt) blocking network access for the `git` process, which Go's tooling invokes internally.
-   **Attempted Mitigations:**
    1.  Configuring `GOPATH` and `GOMODCACHE` to be project-local.
    2.  Using `GOPROXY` to force module downloads over HTTPS.
-   **Result:** These attempts failed because the Go toolchain still fell back to a direct `git` call to resolve module versions. Even after the user reported disabling the sandbox, the issue persisted, indicating a potentially complex environmental or configuration problem.

**A key requirement for this project is to propose a robust and reliable solution for managing Go dependencies within this constrained environment.**

3.  **Binaries:** Cross-compiled executables for the target platforms.
