# ドキュメント作成ガイド：lite-rag 向け Markdown の書き方

このガイドでは、lite-rag がドキュメントをどのように処理するかを説明し、
検索品質を最大化するための Markdown 記述のベストプラクティスを紹介します。

---

## lite-rag がドキュメントを処理する流れ

パイプラインを理解することで、インデックス効率の良いドキュメントを書けるようになります。

```
Markdown ファイル
      │
      ▼
  正規化              Unicode NFKC、空白クリーンアップ、改行コード統一
      │
      ▼
  見出しで分割         見出しごとに新しいセクションを生成。
      │               HeadingPath を累積構築（例：「# ガイド > ## インストール > ### Linux」）
      ▼
  アトム化            セクションを段落に分割。段落が chunk_size を超える場合は
      │               文末（日本語：。、英語：./?/!）でさらに分割
      ▼
  チャンクに詰め込む  段落・文を chunk_size トークンまで結合。
      │               chunk_overlap トークン分を次チャンクに持ち越す
      ▼
  埋め込み・保存      各チャンクにベクトル埋め込みを付与し、HeadingPath・本文・
                      ドキュメントメタデータとともに DuckDB に保存
```

チャンカーが使うトークン数推定:

- CJK 文字（漢字・ひらがな・カタカナ）: **1 文字 ≈ 2 トークン**
- ASCII・ラテン文字の単語: **1 単語 ≈ 1.3 トークン**

デフォルト設定: `chunk_size = 512`、`chunk_overlap = 64`

---

## 1. 見出しでセマンティックなセクションを作る

見出しは lite-rag がドキュメント構造を理解する主要な手がかりです。
各チャンクはそのセクションの完全な見出し階層を `HeadingPath` として保持します。
このパスは検索結果に表示され、LLM プロンプトのコンテキストとして使われます。

**良い例 — 明確な階層:**

```markdown
# インストールガイド

## macOS

Homebrew をインストールしてから以下を実行します:

```sh
brew install lite-rag
```

## Linux

リリースページからバイナリをダウンロードします。
```

`brew install` の段落には `HeadingPath = "# インストールガイド > ## macOS"` が付くため、
macOS のインストールに関する質問でこのチャンクが高スコアでヒットします。

**避けるべき例 — フラットな構造:**

```markdown
# すべて

セットアップ・設定・使い方・トラブルシューティングを以下に説明します。

brew でインストール。config.toml で設定。lite-rag ask "質問" で実行。
問題があればログを確認。
```

サブ見出しがないと、無関係なトピックが同一の大きなセクションに詰め込まれ、
各トピックの検索精度が下がります。

## 2. 各セクションは一つのトピックに絞る

チャンカーはセクションをトピックで分割できません。サイズによってのみ分割します。
複数の関心事が混在したセクションは、どの単一クエリに対しても類似度スコアが低下します。

**良い例:**

```markdown
## 設定

`config.toml` を編集して埋め込みモデルと API エンドポイントを設定します。
`chunk_size` フィールドはチャンクのトークン数上限を制御します（デフォルト: 512）。
`chunk_overlap` フィールドはチャンク境界をまたぐ文脈の継続性を確保します（デフォルト: 64）。
```

**避けるべき例:**

```markdown
## その他

config.toml を編集する。index コマンドはディレクトリを再帰的に走査する。
lite-rag ask で質問できる。データベースは DuckDB に保存される。
コミット前に make check を実行する。
```

## 3. 宣言的に書く — 事実を明示する

埋め込みモデルはクエリベクトルとドキュメントベクトルを比較します。
クエリは疑問文であることが多いですが、ドキュメントは答えを含みます。
ドキュメントが事実を直接・明示的に述べているとき、マッチが最も強くなります。

**良い例:**

```markdown
デフォルトの埋め込みモデルは `nomic-embed-text-v1.5` です。
lite-rag はベクトル埋め込みをローカルの DuckDB データベースファイル（`lite-rag.db`）に保存します。
ファイルハッシュと埋め込みモデルの両方が保存済みの値と一致する場合、再インデックスはスキップされます。
```

**避けるべき例:**

```markdown
前述のとおり、モデルは変更できます。データベース関連は自動で処理されます。
以前に実行済みであれば、再実行はされません。
```

曖昧な代名詞・受動態・前後参照はすべてセマンティックマッチングを弱めます。

## 4. セクションのサイズを chunk_size 以内に収める

デフォルトの `chunk_size = 512` トークンでは、日本語約 200〜250 文字または
英語約 300〜500 単語が 1 チャンクに収まります。セクションがそれより大きい場合、
チャンカーは段落または文境界で分割します。

分割自体は問題ではありませんが、1 つの事実が 2 つのチャンクに分かれると、
片方のチャンクにしかその事実が現れなくなる場合があります。
独立した事実は 1 つの段落に収めることを推奨します。

**目安:**

| コンテンツ種別 | 1 チャンクに収まる量 |
|--------------|-------------------|
| 日本語散文 | 約 200〜250 文字 |
| 英語散文 | 約 300〜400 単語 |
| 日英混在 | 日本語約 150〜200 文字 + 英語約 100 単語 |
| コードブロック | トークン数に含まれる；スニペットは簡潔に |

---

## 5. 段落間には空白行を入れる

チャンカーはセクションを**2 つ以上の連続した改行**（空白行）で段落に分割します。
空白行のない文章の塊は 1 段落として扱われ、まとめてパックされます。
異なる考えは空白行で区切り、チャンカーが独立して処理できるようにしてください。

**良い例:**

```markdown
## アーキテクチャ

lite-rag はベクトルストアとして DuckDB を使用します。埋め込みは FLOAT[768] 列に保存され、
類似度検索は `list_cosine_similarity` で実行されます。

チャンカーは見出しと段落境界で Markdown を分割します。長い段落は
chunk_size 以内に収まるよう文末でさらに分割されます。
```

**避けるべき例:**

```markdown
## アーキテクチャ
lite-rag はベクトルストアとして DuckDB を使用します。埋め込みは FLOAT[768] 列に保存され類似度検索は list_cosine_similarity で実行されます。チャンカーは見出しと段落境界で Markdown を分割します。長い段落は chunk_size 以内に収まるよう文末でさらに分割されます。
```

## 6. 日本語：文末には 。を使う

段落が `chunk_size` を超える場合、チャンカーは日本語の文境界（。）で分割します。
。で終わらない文は末尾フラグメントとして扱われ、まとめて保持されるため、
チャンクが大きくなる場合があります。

**良い例:**

```markdown
デフォルトのチャンクサイズは512トークンです。
オーバーラップは64トークンに設定されています。
これにより、チャンク境界をまたぐ文脈が保持されます。
```

**避けるべき例:**

```markdown
デフォルトのチャンクサイズは512トークンで、オーバーラップは64トークンに設定されており
これにより、チャンク境界をまたぐ文脈が保持されます
```

## 7. コードはすべてフェンスコードブロックで囲む

フェンスコードブロック（` ``` ` または `~~~`）内のコンテンツはプレーンテキストとして扱われ、
`#` で始まる行は見出しとして解析されません。コードはチャンクの本文に含まれ、
トークン数にも算入されます。

コードスニペットは短く、直接関連する本文の近くに置いてください。
長いコードブロックは、説明的な見出しで複数のセクションに分割することを検討してください。

```markdown
## サーバーの起動

以下のコマンドで組み込みサーバーを起動します:

```sh
lite-rag serve --port 8080
```

サーバーはデフォルトで `localhost:8080` でリッスンします。
```

## 8. 本文なしの見出しを避ける

別の見出しが直後に続く見出しは空のセクションを生成します。
空のセクションはチャンカーによって無視されます。
直後の見出しの本文には最初の見出しのコンテキストが引き継がれません。

**避けるべき例:**

```markdown
# ガイド
## インストール
## 使い方

使い方の説明をここに書く。
```

`## インストール` セクションは空のため削除されます。
各見出しには最低限の説明文を追加してください。

---

## 9. 用語を統一する

埋め込みモデルは意味的に類似したテキストをマッチさせますが、
同じ概念に異なる用語を使うとドキュメント間の横断検索精度が下がります。

| 不統一な例 | 統一した例 |
|-----------|-----------|
| 「ベクトル DB」「埋め込みストア」「データベース」 | 常に「DuckDB ベクトルストア」 |
| 「チャンクサイズ」「ブロック長」「トークン上限」 | 常に「chunk_size」 |
| 「再インデックス」「インデックス更新」「リフレッシュ」 | 常に「再インデックス」 |

## 10. チェックリスト

- [ ] 主要なトピックごとに見出し（H2 または H3）がある
- [ ] 各セクションは一つのトピックに絞られている
- [ ] 事実が明示的・宣言的に書かれている
- [ ] 段落間に空白行が入っている
- [ ] 日本語の文末に 。がついている
- [ ] コードは言語識別子付きのフェンスコードブロックに入っている
- [ ] 見出しの直後に別の見出しがない（空のセクションがない）
- [ ] ドキュメントセット全体で用語が統一されている
