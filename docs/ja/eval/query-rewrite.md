## v2 評価（2026-03-21）— docs/ データセット

### 概要

プロジェクト全ドキュメント（`docs/`、16 ファイル、165 チャンク）を対象に 9 種類の
ベンチマーククエリを実行した。クエリリライトなし（ベースライン）と LLM によるリライトあり
（ハイブリッド）を比較した。

評価データベースは `testdata/db/lite-rag-docs-20260321.db` に保存した。
このデータベースはリリース直前時点のドキュメントから生成したものであり、
`docs/eval/` も含まれるが、そこに記載された内容は v1 評価の結果である点に留意すること。

### 評価指標の定義

| 指標 | 説明 |
|------|------|
| Top Score | 上位 1 パッセージのコサイン類似度 |
| Mean Score | 返却された全パッセージの平均コサイン類似度 |
| Recall | 期待ファイルがいずれかのパッセージに含まれているか（out_of_scope はスコア < 0.60 を「正解」とする）|
| Latency (ms) | `Retrieve()` 全体のレイテンシ（埋め込み・LLM リライト・ベクトル検索を含む）|

### 分析

#### スコア改善（8 クエリ中 7 件）

スコアが 0.005 以上改善したクエリは 8 件中 7 件（88%）。v1 の 5/7（71%）から向上した。
改善幅が大きかった上位 3 件:

- **chunk_size**: +0.095（0.704 → 0.799）— 両評価で最大の改善幅。「何トークンですか？」という
  質問文が `architecture.md` の TOML スニペット付近の記述に近い宣言文にリライトされたと推察される。
- **overlap / idempotent / quality_gate**: それぞれ +0.047 — 技術的・手続き的クエリで一貫した改善。
- **serve_cmd**: +0.036（0.711 → 0.747）— 改善はあるが Recall は ✗。
  serve サブコマンドの説明が README に短く言及されるのみで、専用ドキュメントがない。

引き分けだった vector_db（0.750 = 0.750）は Recall が ✗ → ✓ に改善しており、リライトが
別のパッセージを上位に引き上げた可能性がある。

#### ソース再現率（ベース 3/9 → リライト 5/9）

リライトにより再現率が 33% → 56% に向上した。依然として失敗した 4 件:

1. **overlap**: 期待する `architecture.md` が上位に浮上せず。overlap セクションが
   アーキテクチャ文書の一部として短くまとめられており、`context_window` を増やすと改善する見込み。
2. **quality_gate**: `setup.md` には「make check」の説明があるが「品質ゲート」という表現が
   使われていない。オーサリングガイドが推奨する「検索されるキーワードを本文に含める」原則の
   実例と言える。
3. **out_of_scope**: トップスコア 0.732 が低関連性警告の閾値（0.60）を上回っている。
   docs/ データセットはファイル数が多く、無関係なクエリでも類似度の底上げが起きる。
   このデータセット規模には 0.70 前後の閾値が適切かもしれない。
4. **serve_cmd**: README への短い言及のみでは Recall に不十分。専用の使い方ドキュメントが望ましい。

#### v1 との比較（testdata/docs/ データセット）

| 指標 | v1（5 ファイル） | v2（16 ファイル）|
|------|----------------|----------------|
| リライトスコア勝利 | 5/7（71%）| 7/8（88%）|
| リライト再現率 | 5/8（62.5%）| 5/9（56%）|
| ベース再現率 | 5/8（62.5%）| 3/9（33%）|
| 平均レイテンシ増加 | +1,433 ms | +2,178 ms |

v2 で再現率が下がっているのは想定内。16 ファイルに渡る競争的な検索環境では期待ファイルが
上位に浮上しにくくなるが、スコア改善は 5/7 → 7/8 と向上している。レイテンシ増加は
より大きなインデックスに対する検索コスト増加を反映している。

### 推奨設定

| ユースケース | 推奨設定 | 理由 |
|------------|---------|------|
| 精度優先（バッチ・非同期） | `query_rewrite = true` | 88% のクエリでスコア向上、再現率 +23 pp |
| 応答速度優先（対話型） | `query_rewrite = false` | 約 16 ms vs 約 2,200 ms |
| 英語クエリが多い | `query_rewrite = true` | 英語クエリでも再現率維持、レイテンシ増加は許容範囲 |

### 再現方法

```sh
# バイナリをビルド
make build

# docs/ をインデックスして評価データベースを生成
make eval-build-db
# → testdata/db/lite-rag-docs-YYYYMMDD.db と testdata/db/eval-current.db が生成される

# 評価を実行（stdout にマークダウン表を出力）
make eval
```

評価プログラムは `cmd/eval/main.go` に実装されている。`-db` フラグでデータベースパスを、
`-config` フラグで設定ファイルを指定できる。
